# 门面模式

## S（场景）

游戏中有大量分散的基础服务：地图访问（MapRoot）、会话状态（GameSessionState）、资金管理（FactionFundsLedger）、胜负判定（WinConditionService）。UI 层、命令层、AI 层如果各自直接引用这些服务，调用方需要知道所有服务的创建方式和依赖关系。

## T（挑战）

如何给上层提供一个统一的入口，避免调用方直接感知底层有多少个服务？

## A（方案）

`GameStateFacade` 作为单一门面，聚合所有基础服务的操作：

```csharp
public class GameStateFacade : Singleton<GameStateFacade> {
    public MapRoot MapRoot => _mapRoot;
    public GameSessionState Session => _session;

    // 资金
    public int GetFunds(UnitFaction f) => GameFundsService.Instance.GetFunds(f);
    public bool TrySpendFunds(UnitFaction f, int a) => GameFundsService.Instance.TrySpendFunds(f, a);

    // 回合
    public void OnTurnStart(UnitFaction f) => TurnLifecycleService.Instance.HandleTurnStart(f, _mapRoot);

    // 胜负
    public GameConditionResult CheckWinConditions() => GameOverService.Instance.CheckWinConditions(_mapRoot);
}
```

`GameFundsService` 是资金用例服务，委托给单例 `FactionFundsLedger` 读写。`GameOverService` 同样封装胜负归因逻辑。各服务内部都是单例，但通过门面统一暴露，上层只知道 Facade。

## R（收益）

- 调用方只需持有一个引用（Facade），不需要知道底下有多少服务
- 服务替换成本低：改 `GameFundsService` 实现不影响任何调用方
- 统一事件出口：`OnGameOver` 是游戏结束的唯一广播点，任何子系统检测到胜负都必须走 `NotifyGameOver()`
- 新增服务只需在 Facade 加 property，无需改动已有调用方
