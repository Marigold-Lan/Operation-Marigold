# 共享规则层

## S（场景）

战斗规则、占领规则、补给规则、运输规则、经济规则……这些规则在运行时（玩家操作）和 AI 模拟（Anthropic 搜索）中必须完全一致。如果各写一份实现，改一条规则要改两处，迟早分叉。

## T（挑战）

如何让业务规则只写一次，同时服务于运行时和 AI 模拟，且不引入 Unity 依赖？

## A（方案）

在 `Rules/` 目录下，每个规则类同时提供 Runtime 和 Snapshot 两条执行路径。以战斗规则为例：

```csharp
// Runtime 版本：校验并创建攻击数据
public static bool TryCreateStrike(UnitController attacker, UnitController defender, ...) {
    // 校验存活、武器选择、射程 → 封装在静态方法里
}

// 共享版本：AI 快照专用
public static int CalculateSnapshotDamage(AIBoardState board, AIUnitSnapshot attacker, AIUnitSnapshot defender, bool usePrimary) {
    // 从 board.damageMatrix 查伤害系数，和运行时用同一个 DamageResolver
}
```

所有共享规则类都是纯 C# 静态类，不引用任何 MonoBehaviour。规则通过 `TryXxx()` 返回成功/失败，不直接修改状态——修改由调用方执行，共享规则只负责逻辑判定。

关键共享规则：
- `CombatRulesShared`：战斗合法性 + 伤害计算
- `TurnEconomyRulesShared`：回合收入、补给修理
- `CaptureRulesShared`：占领者判定 + 占领进度计算
- `SupplyRulesShared`：补给校验
- `TransportRulesShared`：装载/卸载校验

## R（收益）

- 零分叉：运行时和 AI 用的是同一套规则定义，永远一致
- 纯 C#：`*Shared` 类不依赖 Unity，AI 搜索无 Unity 开销
- 规则复用：共享规则可被 `AIGameLogic`（Anthropic 逻辑）和 `CommandValidator`（交互校验）同时引用
- 易于审计：所有业务规则集中在 Rules/ 目录，改规则时一个文件改到底，不漏不重复
